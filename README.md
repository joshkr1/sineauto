# Sineauto

Web app for Sineauto — vehicle inventory, inquiries, and admin dashboard.

## Tech stack

- **Framework**: [TanStack Start](https://tanstack.com/start) v1 (React 19 + Vite 7, SSR-capable, deploys to Cloudflare Workers / edge)
- **Styling**: Tailwind CSS v4 + shadcn/ui (Radix primitives)
- **Backend**: Supabase (Postgres + Auth + Storage + Edge Functions)
- **Routing**: File-based, auto-generated by `@tanstack/router-plugin`
- **Deploy**: GitHub Actions → Cloudflare (`.github/workflows/deploy.yml`)

## Why route files look "flat" (e.g. `admin.users.tsx`)

This is **TanStack Router's flat dot-separated convention**, not disorganization.
- `admin.tsx` → `/admin` (layout, must render `<Outlet />`)
- `admin.users.tsx` → `/admin/users` (renders inside the admin layout)
- `inventory.$id.tsx` → `/inventory/:id` (dynamic param)

Renaming or moving these files into folders **breaks routing**. The router plugin
auto-generates `routeTree.gen.ts` based on these exact filenames.

See: <https://tanstack.com/router/latest/docs/framework/react/guide/file-based-routing>

## Server functions vs client code

- **`src/utils/*.functions.ts`** — `createServerFn` wrappers. Server-only logic
  (DB writes with service role, admin actions). Safe to import from components —
  the build replaces the body with an RPC stub on the client.
- **`*.server.ts`** files — server-only helpers. Vite blocks these from the client
  bundle (`[import-protection] Import denied in client environment`).
- **Everything else** — runs on both client and server (loaders are isomorphic).

> ⚠️ Never put DB queries or secret keys directly in a route `loader` — loaders
> run on the client too. Wrap them in `createServerFn`.

## Local development

```bash
bun install
bun run dev
```

Required environment variables (in `.env`):
```
VITE_SUPABASE_URL=
VITE_SUPABASE_PUBLISHABLE_KEY=
SUPABASE_URL=
SUPABASE_PUBLISHABLE_KEY=
SUPABASE_SERVICE_ROLE_KEY=     # server-only, never expose to client
```

## Database

Migrations live in `supabase/migrations/`. After changing the schema, regenerate
types with the Supabase CLI so `src/integrations/supabase/types.ts` stays in sync.

## Auth & roles

- Authentication: Supabase Auth (email + Google OAuth).
- Roles: stored in a separate `user_roles` table (NEVER on `profiles`) and checked
  via the `has_role()` security-definer function in RLS policies. This prevents
  privilege-escalation attacks.
